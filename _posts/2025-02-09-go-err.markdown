Boilerplate. The worst programming *thing* ever. I once asked my friend
if he'd rather meet the devil or write boilerplate code, and they answered
that they'd rather write boilerplate code!? Psycho!!! And one language, known
as the pioneer of boilerplate code, is certainly NOT Go. Yet Go has a fundemental
problem in its error handling that causes a *lot* of boilerplate code.

## I: Assessing the problem

Take a look at this python code:

{% highlight python %}
try:
    print("Hello World!");
except Exception:
    print("NOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOO");
{% endhighlight %}

*Yes*, I will say something positive about python for once, this is
*good code*. It's not boilerplate code and it's awesome! Now, take
this Go code:

{% highlight go %}
_, err := fmt.Println("Hello World!");
if err != nil {
    fmt.Println("NOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOOO");
}
{% endhighlight %}

This code is... Fine, but it causes a lot of boilerplate. Essentially,
every time you want to error handle, you have to do this:

{% highlight go %}
_, err := function();
if err != nil {
    do_something_about_it();
}
{% endhighlight %}

THIS IS TERRIBLE!!! It's the only thing that programmers should be
fundamentally against: *boilerplate*. So lets take a look at some
solutions!

## II: try/catch

Perhaps the most common solution to this is the ``try/catch`` error
handling method. Basically:

{% highlight python %}
try:
   # Try to do something, perhaps:
   #     print("Hello World!");
# The conventional approach is:
#     except ERROR:
# but in some languages this is
# allowed.
except:
    # Do something if the ``try`
    # Statement fails, perhaps:
    #     print("ERROR: Something went wrong!");
{% endhighlight %}

This has been highlighted as a solution in [this paper](https://github.com/golang/proposal/blob/master/design/32437-try-builtin.md), and this is the paper summarized (for you lazy people out there):

## III: The try/catch paper

We propose a new built-in function called try, designed specifically to eliminate the boilerplate
if statements typically associated with error handling in Go. No other language changes are suggested.
The try built-in is easy to explain, straightforward to implement, orthogonal to other language constructs,
and fully backward-compatible. It also leaves open a path to extending the mechanism, should we wish to do
so in the future. We propose to add a new function-like built-in called try with signature (pseudo-code):

{% highlight go %}
func try(expr) (T1, T2, … Tn)
{% endhighlight %}

where ``expr`` stands for an incoming argument expression (usually a function call) producing n+1 result
values of types ``T1``, ``T2``, ... Tn, and error for the last value. If expr evaluates to a single value
(n is 0), that value must be of type error and try doesn't return a result. Calling try with an expression
that does not produce a last value of type error leads to a compile-time error. The try built-in may only be
used inside a function with at least one result parameter where the last result is of type error. Calling try
in a different context leads to a compile-time error. Invoking try with a function call f() as in
(pseudo-code):

{% highlight go %}
x1, x2, … xn = try(f())
{% endhighlight %}

turns into the following (in-lined) code:

{% highlight go %}
t1, … tn, te := f()  // t1, … tn, te are local (invisible) temporaries
if te != nil {
        err = te     // assign te to the error result parameter
        return       // return from enclosing function
}
x1, … xn = t1, … tn  // assignment only if there was no error
{% endhighlight %}

## IV: Is try/catch good?

... Not really. I like the idea, I like it a lot, don't get me wrong! It's
just a bit... Problematic. Firstly, it's a function and not a keyword. For
this implementation to even be considered, I'd like it to look more like:

{% highlight go %}
try {
    function_1();
} except {
    function_2();
}
{% endhighlight %}

Or maybe with different error values, like this:

{% highlight go %}
try {
    function_1();
} except StackOverflowError {
    function_2();
}
{% endhighlight %}

But then again, try/catch isn't really a *good* error handling method,
it's ugly and still requires *some* boilerplate. I draw the conclusion
that try/catch *isn't* good. But bonus points for effort!

## V: The fabled "?"

Not too long ago, a [discussion](https://github.com/golang/go/discussions/71460) was opened on the go github repo, proposing
a *new* error handling strategy. It's basically rust's "?"
operator all over again. It's basically:

{% highlight go %}
function_1()?
{% endhighlight %}

## VI: The "?" paper summarized

This section presents the formal proposal.

An assignment or expression statement may be followed by a question mark (?). The question mark is a new syntactic element, the first permitted use of ? in Go outside of string and character constants. The ? causes conditional execution similar to an if statement. A ? at the end of a line causes a semicolon to be automatically inserted after it.

A ? uses a value as described below, referred to here as the qvalue.

For a ? after an assignment statement, the qvalue is the last of the values produced by the right hand side 
of the assignment. The number of variables on the left hand side of the assignment must be one less than the
number of values produced by the right hand side (the right hand side values may come from a function call
as usual). It is not valid to use a ? if there is only one value on the right hand side of the assignment.

For a ? after an expression statement the qvalue is the last of the values of the expression. It is not
valid to use a ? after an expression statement that has no values.

The qvalue must be of interface type and must implement the predeclared type error; that is, it must have
the method Error() string. In most cases it will simply be of type error.

A ? is optionally followed by a block. The block may be omitted if the statement using ? appears in the body
of a function, and the enclosing function has at least one result, and the qvalue is assignable to the last
result (this means that the type of the last result must implement the predeclared type error, and will
often simply be error).

Execution of the ? depends on the qvalue. If the qvalue is nil, execution proceeds as normal, skipping over
the block if there is one.

If the ? is not followed by a block, and the qvalue is not nil, then the function returns immediately. The
qvalue is assigned to the final result. If the other results (if any) are named, they retain their current
values. If they are not named, they are set to the zero value of their type. The results are then returned.
Deferred functions are executed as usual.

If the ? is followed by a block, and the qvalue is not nil, then the block is executed. Within the block a
new variable err is implicitly declared, possibly shadowing other variables named err. The value and type of
this err variable will be those of the qvalue.

That completes the proposal.
